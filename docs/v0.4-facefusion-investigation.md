# AI Studio v0.4 FaceFusion Investigation

## Scope

This document investigates how FaceFusion should be integrated into AI Studio in a future v0.4 phase. It is research and design only. No implementation is included.

Primary sources checked:

- FaceFusion official docs introduction and usage: https://docs.facefusion.io/
- FaceFusion CLI commands: https://docs.facefusion.io/usage/cli-commands/general
- FaceFusion CLI path arguments: https://docs.facefusion.io/usage/cli-arguments/paths
- FaceFusion CLI execution arguments: https://docs.facefusion.io/usage/cli-arguments/execution
- FaceFusion output creation arguments: https://docs.facefusion.io/usage/cli-arguments/output-creation
- FaceFusion processors arguments: https://docs.facefusion.io/usage/cli-arguments/processors
- FaceFusion Docker usage: https://docs.facefusion.io/usage/run-with-docker
- FaceFusion GitHub repository: https://github.com/facefusion/facefusion

Note: the official docs page currently displays documentation version `3.6.0`, while the GitHub repository shows a newer latest release line. Before implementation, pin the exact FaceFusion version to avoid argument drift.

## 1. Official Startup Modes

FaceFusion is officially presented as a command-line application launched with:

```bash
python facefusion.py [commands] [options]
```

The official command list includes:

- `run`: starts the program with UI support.
- `headless-run`: runs in headless mode.
- `batch-run`: runs batch processing.
- `job-*` commands: create, submit, run, retry, and delete jobs.
- `benchmark` and `force-download` support operations.

The installation guide ends with a UI-oriented command:

```bash
python facefusion.py run --open-browser
```

The Docker guide also exposes browser URLs for CPU/CUDA/TensorRT/ROCm containers, which indicates a Web UI runtime, not a first-class HTTP API contract.

Current official surface by category:

| Mode | Officially documented | Notes |
| --- | --- | --- |
| CLI | Yes | Main documented interface. |
| Python API | No stable public API documented | Internal Python modules exist, but they are not documented as an integration API. |
| Web UI | Yes | Started through `run`; useful for humans, not ideal as backend integration. |
| HTTP API | Not found in official docs | No stable REST API contract found in official docs. |

## 2. Best Fit For FastAPI Integration

The best fit for AI Studio is CLI-based headless execution through a `FaceFusionExecutor` subprocess wrapper.

Recommended approach:

```text
FastAPI endpoint
  -> FaceSwapService
      -> TaskManager
      -> FileService
      -> FaceFusionExecutor
          -> subprocess: python facefusion.py headless-run ...
```

Reasons:

- It follows FaceFusion's documented command interface.
- It avoids relying on undocumented internal Python functions.
- It does not require starting or supervising a separate Web UI service.
- It maps cleanly to AI Studio's existing task workflow.
- stdout, stderr, return code, timeout, GPU provider, input paths, and output path can be captured per task.

The Web UI is not recommended for FastAPI integration because it is designed for browser interaction. A hypothetical internal HTTP route behind the UI would be less stable than the CLI unless the official project documents it.

## 3. Minimal FaceFusion Call Flow

For a single face-swap task, the minimal documented inputs are:

- Source image/audio paths via `--source-paths`.
- Target image/video path via `--target-path`.
- Output file path via `--output-path`.
- Processor selection, with `face_swapper` as the documented default processor.
- Execution provider/device selection when GPU control is needed.

Conceptual command:

```bash
python facefusion.py headless-run \
  --source-paths /path/to/source.png \
  --target-path /path/to/target.png \
  --output-path /path/to/output.png \
  --processors face_swapper \
  --execution-providers cuda \
  --execution-device-ids 0
```

The official docs define these relevant arguments:

- `--source-paths`: image or audio inputs.
- `--target-path`: image or video target.
- `--output-path`: output image or video path.
- `--processors`: one or more processors, including `face_swapper` and `face_enhancer`.
- `--execution-providers`: `cpu`, `cuda`, `tensorrt`, `rocm`, and others.
- `--execution-device-ids`: device IDs used for processing.

## 4. Backend Service Inputs And Outputs

As an AI Studio backend capability, FaceFusion should not expose raw FaceFusion CLI arguments directly to users.

Recommended AI Studio input:

```json
{
  "source_file_id": "file_source",
  "target_file_id": "file_target",
  "options": {
    "processor": "face_swapper",
    "enhance_face": false,
    "execution_provider": "cuda",
    "execution_device_id": 0,
    "output_format": "png"
  }
}
```

FaceFusionExecutor resolves this into:

```text
source_file_id -> source path from FileService
target_file_id -> target path from FileService
output path    -> generated path under data/outputs/{task_id}/...
options        -> validated CLI arguments
```

Recommended output:

```json
{
  "task_id": "task_...",
  "status": "succeeded",
  "result": {
    "file_id": "file_output",
    "download_url": "/api/v1/tasks/task_.../result"
  }
}
```

The executor should return an output artifact path or output file metadata to the service layer. It should not return raw CLI logs as the primary API result. Logs should be stored separately for debugging.

## 5. Can FaceFusion Be Called Directly In Python?

Technically, FaceFusion is written in Python and has internal modules such as `facefusion.core`. However, the official docs do not present a stable Python SDK/API for external application integration.

Direct Python imports are not recommended for AI Studio v0.4 because:

- The public contract is CLI-oriented.
- Internal modules use global state management.
- CLI routing can call process exits.
- Internal function signatures may change between FaceFusion releases.
- Importing into the FastAPI process risks dependency, GPU memory, and global state conflicts.

Recommended decision: treat FaceFusion as an external executable and call it from `FaceFusionExecutor` via subprocess.

## 6. Is An Independent Service Required?

No. For backend processing, an independent long-running FaceFusion service is not required if we use `headless-run` per AI Studio task.

Possible deployment modes:

| Mode | Requirement | Recommendation |
| --- | --- | --- |
| Subprocess per task | No separate service | Best first integration. |
| Web UI service | Requires running FaceFusion UI | Not recommended for backend integration. |
| Docker UI service | Requires container and browser-facing port | Useful for manual testing, not the first API integration. |
| Custom FaceFusion microservice | Requires extra service ownership | Consider later if subprocess startup cost is too high. |

The subprocess model is simplest and aligns with AI Studio's `TaskManager + Executor` architecture.

## 7. GPU Selection

FaceFusion uses execution providers and device IDs.

Relevant official arguments:

```bash
--execution-providers cuda
--execution-device-ids 0
--execution-thread-count 4
```

Provider choices documented include:

- `cpu`
- `cuda`
- `tensorrt`
- `rocm`
- `directml`
- `coreml`
- `migraphx`

For AI Studio, recommended config fields:

```text
FACEFUSION_EXECUTION_PROVIDER=cuda
FACEFUSION_EXECUTION_DEVICE_IDS=0
FACEFUSION_EXECUTION_THREAD_COUNT=4
```

If multiple GPUs are available, AI Studio should not let arbitrary users choose device IDs directly. Instead, map user tasks to configured executor queues or safe backend configuration.

## 8. Output Directory Configuration

FaceFusion supports output path configuration through:

```bash
--output-path /path/to/output.png
```

It also supports temp and jobs directories:

```bash
--temp-path .temp
--jobs-path .jobs
```

For AI Studio, output paths should be generated by the backend, not accepted directly from users.

Recommended mapping:

```text
AI Studio input files:   data/uploads/{file_id}/...
AI Studio output files:  data/outputs/{task_id}/result.png
FaceFusion temp path:    data/runtime/facefusion/temp/{task_id}
FaceFusion jobs path:    data/runtime/facefusion/jobs
```

After FaceFusion completes, `FileService` should register the generated output and expose it through:

```http
GET /api/v1/tasks/{task_id}/result
```

## 9. Official Directory Structure

The official repository is a standalone application layout, not a Python package SDK layout. The GitHub repository includes top-level files such as:

```text
facefusion.py
install.py
facefusion.ini
requirements.txt
facefusion/
tests/
```

The official docs also use a root-level `facefusion.ini` as the default config path and mention `.jobs` as the default jobs directory.

For AI Studio, we should not merge FaceFusion source into `backend/app`. Keep it external and configurable:

```text
/opt/facefusion/                 # cloned or installed FaceFusion runtime
/3241903007/workstation/LYJ/AI-Studio/backend/app/  # AI Studio backend
/3241903007/workstation/LYJ/AI-Studio/data/         # AI Studio files/results/runtime
```

Recommended AI Studio config:

```text
FACEFUSION_ROOT=/opt/facefusion
FACEFUSION_PYTHON=/opt/conda/envs/facefusion/bin/python
FACEFUSION_CONFIG_PATH=/opt/facefusion/facefusion.ini
```

This avoids coupling AI Studio backend code to FaceFusion repository internals.

## 10. Recommended Integration For AI Studio

Recommended v0.4 integration: external CLI adapter using `headless-run`.

Architecture:

```text
POST /api/v1/face-swap/tasks
  -> FaceSwapService
      -> TaskManager marks running
      -> FileService resolves source and target files
      -> FaceFusionExecutor builds command
      -> subprocess.run([...], cwd=FACEFUSION_ROOT, timeout=...)
      -> FileService registers output
      -> TaskManager marks succeeded or failed
```

`FaceFusionExecutor` should extend the existing `BaseExecutor` from v0.3.

Proposed executor responsibilities:

- Build safe argument list, never shell-concatenate user input.
- Run FaceFusion from configured `FACEFUSION_ROOT`.
- Use configured Python interpreter or command path.
- Set `--source-paths`, `--target-path`, `--output-path`.
- Set `--processors face_swapper` initially.
- Set GPU options from server config.
- Capture stdout/stderr and return code.
- Enforce timeout.
- Verify output file exists before marking success.

Proposed config additions:

```text
facefusion_root: Path
facefusion_python: Path
facefusion_config_path: Path | None
facefusion_execution_provider: str = "cuda"
facefusion_execution_device_ids: list[int] = [0]
facefusion_execution_thread_count: int = 4
facefusion_timeout_seconds: int = 600
```

Implementation should keep the current API contract unchanged:

```http
POST /api/v1/files
POST /api/v1/face-swap/tasks
GET /api/v1/tasks/{task_id}
GET /api/v1/tasks/{task_id}/result
```

## Direct Answers

1. Current official startup recommendation: CLI is the primary documented interface; Web UI is started through `run`; Docker starts UI containers; no official Python API or HTTP API was found.
2. Best FastAPI integration: CLI `headless-run` wrapped by `FaceFusionExecutor` subprocess.
3. Minimal call flow: source path + target path + output path + face_swapper processor + execution provider/device.
4. Backend inputs/outputs: input file IDs and validated options in; generated output file metadata and task result out.
5. Direct Python call: technically possible but not recommended because no stable official Python API is documented.
6. Independent service: not required for first integration; use subprocess per task.
7. GPU selection: use `--execution-providers cuda` and `--execution-device-ids 0`, configured server-side.
8. Output directory: generate backend-owned `data/outputs/{task_id}/...` and pass it via `--output-path`.
9. Official directory structure: standalone FaceFusion repo with `facefusion.py`, `facefusion.ini`, `facefusion/`, `.jobs`; AI Studio should keep FaceFusion external.
10. Recommended AI Studio approach: maintain `FileService + TaskManager + BaseExecutor`, add `FaceFusionExecutor` as an external CLI adapter using headless mode.

## Risks And Follow-Up Checks

- Pin the exact FaceFusion version before implementation.
- Confirm installed FaceFusion CLI arguments with `python facefusion.py headless-run --help` on the target server.
- Confirm CUDA provider availability after installation.
- Run one manual command with a known source/target pair before wiring it into AI Studio.
- Decide whether FaceFusion should use a dedicated conda environment instead of AI Studio's `df` environment.
- Store stderr/stdout logs per task for debugging failed runs.
