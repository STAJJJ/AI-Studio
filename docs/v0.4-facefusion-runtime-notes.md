# AI Studio v0.4 FaceFusion Runtime Notes

## Purpose

Stage 4 validates that AI Studio can call FaceFusion through a subprocess-backed `FaceFusionExecutor`. This stage does not connect FaceFusion to API, TaskManager, FileService, Workflow, frontend, or ComfyUI.

## Official FaceFusion Version

The FaceFusion repository was cloned from:

```text
git@github.com:facefusion/facefusion.git
```

At the time of verification, the cloned repository default branch pointed to:

```text
3.7.1
```

The official documentation presents FaceFusion as a CLI-driven application with commands such as:

```bash
python facefusion.py run
python facefusion.py headless-run
python facefusion.py force-download
```

## Current Runtime Version

AI Studio's stage 4 verification used FaceFusion:

```text
3.0.0
```

The external runtime path is:

```text
/3241903007/workstation/LYJ/FaceFusion/facefusion
```

FaceFusion is intentionally kept outside the AI Studio repository so the backend codebase does not vendor or couple itself to FaceFusion internals.

## Why 3.7.1 Was Not Used

FaceFusion `3.7.1` could not start inside the current `df` conda environment because `df` uses Python `3.10.20`, while FaceFusion `3.7.1` imports newer typing features and dependencies that require Python `3.11+`.

Observed blocker:

```text
ImportError: cannot import name 'NotRequired' from 'typing'
```

The `3.7.1` dependency set also includes packages that do not resolve cleanly on Python 3.10, such as newer SciPy versions requiring Python 3.11 or later.

## Python Compatibility

Current AI Studio environment:

```text
conda environment: df
python: 3.10.20
```

FaceFusion compatibility notes:

- FaceFusion `3.0.0` can run on Python 3.10 with its matching dependency set.
- FaceFusion `3.7.1` should be treated as Python 3.11+ for practical integration.
- A future production integration should use a dedicated FaceFusion runtime environment instead of sharing AI Studio's `df` environment.

Recommended future runtime:

```text
/opt/conda/envs/facefusion
Python 3.11+
FaceFusion pinned version
```

AI Studio can still call that runtime through subprocess by setting the executor's `python_executable` to the FaceFusion environment's Python binary.

## CPU Provider Verification

The stage 4 integration test used CPU execution because the installed FaceFusion dependency set used default `onnxruntime`, not `onnxruntime-gpu`.

Verified command shape:

```bash
python facefusion.py headless-run \
  --source-paths /3241903007/workstation/LYJ/AI-Studio/backend/tests/assets/source.png \
  --target-path /3241903007/workstation/LYJ/AI-Studio/backend/tests/assets/target.png \
  --output-path /3241903007/workstation/LYJ/AI-Studio/backend/tests/assets/output.png \
  --processors face_swapper \
  --execution-providers cpu \
  --execution-device-id 0
```

Verification result:

```text
tests/test_facefusion_executor.py passed
output file: backend/tests/assets/output.png
```

## Integration Test Policy

FaceFusion CLI verification is an integration test. It is intentionally excluded from default test runs because it can take several minutes and depends on external runtime files, model downloads, and ffmpeg.

Default regression:

```bash
python -m pytest -q
```

Run FaceFusion integration explicitly:

```bash
RUN_FACEFUSION_TEST=1 python -m pytest tests/test_facefusion_executor.py -q -s
```

The test is also marked with:

```python
@pytest.mark.integration
```

## GPU Provider Migration Plan

Future GPU validation should not mutate AI Studio's `df` environment. Use a dedicated FaceFusion environment.

Recommended steps:

1. Create or provision a dedicated Python 3.11+ FaceFusion environment.
2. Install FaceFusion's CUDA dependency variant, such as `onnxruntime-gpu`, through FaceFusion's installer or pinned requirements.
3. Verify GPU provider availability with FaceFusion CLI help and a small headless run.
4. Update AI Studio config to point to the dedicated FaceFusion Python executable.
5. Change executor settings from CPU to CUDA:

```text
execution_provider=cuda
execution_device_ids=[0]
```

6. Keep the AI Studio API and task workflow unchanged. Only the executor runtime configuration should change.

## Current Recommendation

Keep stage 4 as proof of CLI integration only. Before connecting FaceFusion to the production task workflow, isolate FaceFusion into its own runtime environment and pin the exact FaceFusion version.
